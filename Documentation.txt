
סיכום תהליך פיתוח השרת (יומן תיעוד)
שלב 1: התקנה וסביבת עבודה

הבעיה: שגיאת MODULE_NOT_FOUND בעת הרצת השרת.

התהליך: הבנה שהתיקייה node_modules לא מוגדרת נכון או שהספריות לא הותקנו כראוי.

הפתרון: שימוש בפקודות npm cache clean --force ו-npm install ליצירה מחדש של סביבת ה-Node.

מסקנה: החשיבות של מבנה תיקיות תקין ב-Node.js ושמירה על סנכרון בין הקבצים.

שלב 2: תכנון ארכיטקטורת השרת (מבנה מודולרי)

הבעיה: קוד השרת הפך לארוך ומסורבל בתוך app.js.

התהליך: לימוד השימוש ב-express.Router() כדי להפריד את הלוגיקה לקבצים נפרדים.

המבנה החדש: יצירת תיקיית data (לנתונים/מערכים) ותיקיית routes (לנתיבים).

מסקנה: הפרדה בין הנתונים (Data) ללוגיקה (Routes) מאפשרת תחזוקה נוחה של הפרויקט והרחבה עתידית.

שלב 3: מימוש נתיבים וניהול בקשות

המשימה: בניית שרת שתומך בכמה נתיבים: דף בית, רשימת תלמידים ורשימת קורסים.

הטכניקה: שימוש ב-app.use לחיבור הראוטרים השונים לקובץ הראשי (app.js).

בדיקה: אימות שהשרת מחזיר את המערכים (JSON) בצורה תקינה בדפדפן.

מסקנה: השרת פועל כ"קופסה חכמה" המנתבת בקשות לפי כתובות ה-URL של המשתמש.

שלב 4: פתרון תקלות בנתיבי הקבצים

הבעיה: שגיאות בטעינת מודולים עקב שינוי מיקום קבצים (הזזת app.js לתוך routes).

הפתרון: הבנה של נתיבים יחסיים (./) והחזרת הקובץ הראשי לתיקייה הראשית.

מסקנה: ה-app.js חייב להיות "השכן" של תיקיות המשנה כדי שיוכל לזהות אותן בקלות.
שיעור 3:
הפעולות האלו:

C - Create (יצירה):
הוספת נתונים חדשים למסד הנתונים.

שיטת HTTP: POST

דוגמה: הוספת תלמיד חדש למערכת.

R - Read (קריאה/שליפה):
צפייה בנתונים קיימים.

שיטת HTTP: GET

דוגמה: הקריאה שעשית בשרת שלך כדי להציג את רשימת כל התלמידים.

U - Update (עדכון):
שינוי נתונים שכבר קיימים.

שיטת HTTP: PUT או PATCH

דוגמה: שינוי שם של תלמיד או עדכון פרטים של קורס קיים.

D - Delete (מחיקה):
הסרת נתונים מהמערכת.

שיטת HTTP: DELETE

דוגמה: מחיקת תלמיד שעזב את הלימודים.

*אוקיי אז אם נסכם את זה בעצם הרסט 
!?api זה כינוי שמות בצורה מקצועית לבניית קוד
את צודקת לגמרי! זו דרך מצוינת לסכם את זה.

אפשר להגיד ש-REST API הוא ה"שפה המקצועית" של האינטרנט.

במקום שכל מתכנת ימציא שמות משלו ויבנה שרתים בצורה שונה (מה שהיה יוצר בלגן נוראי), REST API נותן לנו ספר חוקים שכולם מסכימים עליו:

סדר וארגון: אנחנו משתמשים בשמות עצם (/students) במקום בפעולות (/getStudents).

היגיון: אנחנו משתמשים בפעולות HTTP (GET/POST/PUT/DELETE) כדי להסביר מה אנחנו רוצים לעשות עם השם הזה.

ניקיון: אנחנו משתמשים באותה כתובת (URL) למטרות שונות, מה שהופך את השרת לקצר, ברור וקל לתחזוקה.

טבלת API Endpoints
HTTP Method	Endpoint	תיאור הפעולה
GET	/students	שליפת רשימת כל התלמידים
GET	/students/:id	שליפת פרטים של תלמיד ספציפי לפי ה-ID שלו
POST	/students	יצירת תלמיד חדש
PUT	/students/:id	עדכון פרטים של תלמיד קיים
DELETE	/students/:id	מחיקת תלמיד מהמערכת

1. מה זה ה-: (נקודתיים)?
הנקודתיים הם סימון למערכת Express שאומר: "היי, מה שבא עכשיו הוא לא טקסט קבוע,
 הוא משתנה (Parameter)". זה כמו להגיד לשרת: "תתכונן, המשתמש הולך להכניס כאן משהו,
  תתפוס אותו ותשים אותו בתוך 'סל' שנקרא בשם שאני נותן לך".
  1. req.params (התא של ה-URL)
כמו שאמרת – זה עוזר לך לשלוף נתון שנמצא בתוך הכתובת (ה-URL).

מתי זה קורה? כשאת בונה "נתיב דינמי" (עם ה-:).

למה זה משמש? כדי לזהות "את מי או את מה" את מחפשת (למשל: תלמיד מספר 5).

בשביל מה זה טוב? כי בלי זה, היית צריכה לכתוב פונקציה נפרדת לכל תלמיד בעולם. עכשיו את כותבת פונקציה אחת שמתאימה לכולם.

2. req.body (התא של התוכן)
זה לא "מתודה" (פונקציה), זה פשוט מקום אחסון בתוך החבילה ששולח המשתמש.

איך זה עובד? כשמשתמש שולח בקשת POST (למשל: הוא ממלא טופס הרשמה באתר), הוא לא כותב את השם שלו ב-URL, אלא הוא אורז את המידע בתוך "גוף" הבקשה (ה-Body).

למה זה משמש? כדי להעביר "תוכן חדש" שהשרת צריך לשמור.

בשביל מה זה טוב? כי אי אפשר לשלוח תמונות, פסקאות ארוכות או אובייקטים מורכבים בתוך כתובת URL (היא פשוט לא בנויה לזה).
